## BLOCKERS

### BLOCKER 1 — Generated plugin layout is wrong: plan puts plugin.json at `_generated/<name>/plugin.json` but current layout uses `_generated/<name>/.claude-plugin/plugin.json`

**Location in plan:** Generator Algorithm section, `_write_plugin` function sketch; Architecture Overview output diagram; Validation Strategy "source path" row.

**Issue:** The plan's `_write_plugin` writes to `_generated/<prefix>-<name>/plugin.json` directly:

```python
(out_dir / "plugin.json").write_text(_to_json(plugin_json))
```

But the actual committed generator (`gen_skill_plugins`, `gen_rule_plugins`, `gen_skill_bundles`, etc.) writes to `_generated/<name>/.claude-plugin/plugin.json`. This is confirmed empirically: `_generated/skill-telegram-notify/.claude-plugin/plugin.json` exists and is the live file. The Compat workflows reference `example-mcp@dgxsparklabs-marketplace` and resolve the install through this `.claude-plugin/` subdirectory.

**Why it's blocking:** If the implementing agent follows the plan's code sketch literally, every generated plugin directory will be missing the `.claude-plugin/` subdirectory. Claude Code's `/plugin install` reads `.claude-plugin/plugin.json`, not a top-level `plugin.json`. The entire marketplace would become uninstallable. The `_write_marketplace_json` function would also fail to find any plugin files.

**Suggested resolution:** Change the plan's `_write_plugin` to write `out_dir / ".claude-plugin" / "plugin.json"` (matching current behavior). Update the Architecture diagram output path to `_generated/<prefix>-<name>/.claude-plugin/plugin.json`. Do the same for `_write_bundle_plugin`. The test sketch in `TestGeneratedPlugins.test_individual_plugins_exist` must also reference the `.claude-plugin/plugin.json` path.

### BLOCKER 2 — Skill plugins copy entire source directory; plan replaces copy with in-place plugin.json generation

**Location in plan:** Code Design > Sample Construct Implementations, `SkillConstruct.build_plugin_json`; Generator Algorithm > Phase 1.

**Issue:** The current `gen_skill_plugins` does a `shutil.copytree(skill_dir, target)` and then places `plugin.json` inside the copy. The generated `_generated/skill-telegram-notify/` directory therefore contains the full skill source tree (SKILL.md, scripts/, references/, etc.) plus `.claude-plugin/plugin.json`. The plan's `SkillConstruct.build_plugin_json` returns a dict and the main loop calls `_write_plugin(construct.prefix, name, plugin_json)` — which only writes a single JSON file. There is no `shutil.copytree` equivalent in the new plan.

Claude Code skill plugins require the skill content to be present in the plugin directory (the `skills: ["./"]` field points at the installation directory's root). If only `plugin.json` is written with no actual SKILL.md alongside it, the installed skill will have no content.

**Why it's blocking:** Every skill plugin generated by the new code would install but provide nothing — no SKILL.md, no scripts, no commands. The marketplace appears to work (plugin installs, appears in list) but is actually empty. This is a silent regression the snapshot-diff check would catch in principle, but the plan gives the implementing agent no indication that the copytree step belongs in `build_plugin_json` (or in a separate `_write_plugin` that also copies source content).

**Suggested resolution:** The plan needs to specify clearly that for constructs where generated output = source content + plugin.json wrapper (currently skills and rules), `build_plugin_json` is insufficient — the generator must also copy source content. Two viable options:

- Add a `source_copy_mode` attribute to the Construct protocol (`"copy"` vs `"json_only"`) and handle it in `_write_plugin`.
- Or: split `build_plugin_json` into `build_plugin_json` (returns dict) and `emit(name, target_dir)` (writes everything including copy). This would make `emit` parallel to the Platform protocol.

Rules have an even more complex case: they don't do a full copytree — they copy only `rule.md`, generate `activate.sh`, and create a `rules/` subdirectory. This logic belongs somewhere in the new Construct class but is entirely absent from the plan's code sketch for `RuleConstruct`.

### BLOCKER 3 — `_write_marketplace_json` aggregates `_generated/*/plugin.json` but actual marketplace.json uses a richer per-entry schema

**Location in plan:** Generator Algorithm section, `_write_marketplace_json` function.

**Issue:** The current `gen_marketplace_json` builds each plugin entry as a dict with `name`, `source`, `description`, `version`, `author`, and `category` — all populated by the per-generator functions that already know those values. The plan's `_write_marketplace_json` re-reads `_generated/*/plugin.json` and constructs a stripped entry with only `name` and `source`:

```python
plugins.append({
    "name": plugin_dir.name,
    "source": f"./_generated/{plugin_dir.name}",
    # ... other manifest fields per Claude Code spec
})
```

The `# ... other manifest fields per Claude Code spec` comment is a placeholder — but it hides the fact that `description`, `version`, `author`, and `category` are needed in the manifest and currently present. These fields exist in the individual `plugin.json` files (so the reimplementation can read them back), but the category field is not part of any Claude Code plugin.json spec — it's a marketplace-internal annotation. The current generator writes `category` only to the marketplace.json entries, not to the individual plugin.json files.

**Why it's blocking:** `category` information (skill, rule, skill-bundle, rule-bundle, example, example-bundle) would be permanently lost in the new generator if the implementing agent follows the plan. There's no way to reconstruct `category` by reading `plugin.json` contents alone — the new bundle schema doesn't use a `category` field, and the construct type is implicit in the plugin name prefix, not in the plugin.json. The aggregation pattern also changes the ordering logic (current: sorted by category then name; new: sorted by directory name only).

More importantly: the Compat workflows call `claude plugin install example-mcp@dgxsparklabs-marketplace` — they resolve plugins by name from marketplace.json. The marketplace.json structure must remain valid for the Claude Code marketplace spec. Omitting `description` and `version` from marketplace entries may violate the spec; `TestGeneratedManifests.test_every_plugin_has_required_fields` currently asserts on `name`, `source`, `version`.

**Suggested resolution:** The plan must specify either (a) that marketplace.json entries are built during the Phase 1/2 loops (passing entry dicts forward to `_write_marketplace_json`) — not reconstructed from the filesystem — or (b) that all needed fields are written to each individual `plugin.json` and the aggregator reads them back. It must also address how `category` survives (or explicitly state it is dropped, if that is intentional). Option (a) matches current behavior exactly.

## IMPORTANT

### IMPORTANT 1 — `supports` is typed as `set[type[Construct]]` but used inconsistently with the instance-based registry

**Location in plan:** Platform Protocol sketch (`supports: set[type[Construct]]`); Generator Algorithm Phase 3 loop; `ClaudeCodePlatform.supports = set(CONSTRUCTS.values())`.

**Issue:** The protocol declares `supports: set[type[Construct]]` (a set of *classes*), but `ClaudeCodePlatform` sets `supports = set(CONSTRUCTS.values())` (a set of *instances*). These are different things. The Phase 3 loop then does:

```python
for construct_cls in platform.supports:
    construct = next(c for c in CONSTRUCTS.values() if isinstance(c, construct_cls))
```

This only works if `supports` contains classes, not instances. `ClaudeCodePlatform` is therefore internally inconsistent with the protocol it claims to implement.

The `isinstance` lookup also fails for `OutputStyleConstruct` because its key in CONSTRUCTS is `"output-style"` (hyphenated) — but `isinstance` dispatches on the Python class, which is fine. However, if two constructs happened to share a base class, the `next(...)` call would non-deterministically return either one.

**Suggested resolution:** Either make all Platform.supports contain classes consistently (fix `ClaudeCodePlatform` to use `{type(c) for c in CONSTRUCTS.values()}` or list all 10 classes explicitly), or change the design to store instances in `supports` and do identity lookup instead of `isinstance`. The ClaudeCodePlatform inconsistency is the immediate bug.

### IMPORTANT 2 — `BundleMember.from_str` parses `"kind_or_construct"` but the catalog example uses `"construct:name"` not `"kind:name"`

**Location in plan:** Bundle Schema section, `BundleMember.from_str`; Catalog Schema (After) example members lists.

**Issue:** `from_str` splits on `:` and assigns the left side to a local called `kind_or_construct`, then uses it as `construct` in `cls(kind="construct", construct=kind_or_construct, name=name)`. But `resolve_dependencies` accesses `m.construct` and uses it as a construct-type prefix:

```python
f"{m.construct}-{m.name}" if m.kind == "construct"
```

This produces `skill-telegram-notify` — correct! However, `BundleMember.kind` is set to `"construct"` for all non-bundle members, regardless of whether the prefix is "skill", "rule", "mcp", etc. The field named `kind` therefore conflates two concepts: the type of reference ("construct" vs "bundle") and the construct type ("skill", "rule", "mcp"). This is confusing but workable.

The real issue: a bundle member entry `"hook:example-hook"` would parse as `kind="construct", construct="hook", name="example-hook"`, producing plugin name `hook-example-hook`. But the catch-all bundle `bundle.hook-all` via `members_from_construct = "hook"` calls `constructs["hook"].prefix` to get `"hook"` and produces `hook-example-hook`. These agree. The ambiguity is purely in field naming — `BundleMember.kind == "construct"` should probably be `"plugin"` to avoid the collision with "construct type". This isn't a bug but the agent will likely write confusing code.

**Suggested resolution:** Rename `BundleMember.kind` to `BundleMember.ref_type` with values `"plugin"` and `"bundle"`, and rename `BundleMember.construct` to `BundleMember.plugin_type` or `prefix`. This disambiguates the schema and makes the code self-documenting.

### IMPORTANT 5 — `_scan_source_dir` is defined in `generate_manifest.py` but used in `bundles.py` via `resolve_dependencies`

**Location in plan:** Bundle Schema section, `Bundle.resolve_dependencies`; Generator Algorithm section, `_scan_source_dir` definition.

**Issue:** `Bundle.resolve_dependencies` in `bundles.py` calls `_scan_source_dir(construct.source_directory)`. But `_scan_source_dir` is shown defined in `generate_manifest.py`. This is a cross-module circular dependency: `bundles.py` would need to import from `generate_manifest.py`, but `generate_manifest.py` imports from `bundles.py`. Python would raise an ImportError.

The platform protocol has the same issue in a different direction: `ClaudeCodePlatform.supports = set(CONSTRUCTS.values())` is written inside `platforms.py`, but `CONSTRUCTS` is defined in `constructs.py`, which `platforms.py` already imports. That part is fine. But `_scan_source_dir` belongs in neither place — it should live in a shared utility module (`scripts/utils.py`) that `generate_manifest.py`, `bundles.py`, and potentially `constructs.py` all import from.

**Suggested resolution:** Add `scripts/utils.py` to the module design, or move `_scan_source_dir` to `constructs.py` (since it operates on `Construct.source_directory`). Specify explicitly which module owns each shared helper. The plan says "5 generic [modules]" in the TL;DR but the current design shows only 4 (`constructs.py`, `platforms.py`, `bundles.py`, `generate_manifest.py`).

### IMPORTANT 6 — `members_from_construct` on a `Bundle` marked `frozen=True` alongside `members: list[BundleMember]` allows both to be set simultaneously, with no validation

**Location in plan:** Bundle Schema section, `Bundle` dataclass; Catalog Schema (After) section.

**Issue:** The `Bundle` dataclass is `frozen=True` and has both `members: list[BundleMember]` and `members_from_construct: str | None`. The `resolve_dependencies` method handles each case with an early return:

```python
if self.members_from_construct:
    return [...]
return [m.construct-... for m in self.members]
```

If a catalog author writes both `members = [...]` AND `members_from_construct = "skill"` in the same `[bundle.*]` block, `members_from_construct` silently wins and `members` is ignored. This is an undetected configuration error. Open Question 10 in the plan notes this but frames it only as "should we validate" — it doesn't commit to a resolution.

**Suggested resolution:** Add explicit validation in `load_bundles`: if a bundle spec sets both `members` and `members_from_construct`, raise a `ValueError` with a clear message. This is a 2-line addition and prevents silent mis-cataloging. Commit to this in the plan before handing to the implementing agent (don't leave it as an open question).

### NICE 3 — `MCP.build_plugin_json` reads `mcpServers` from example's `.claude-plugin/plugin.json` but example-mcp stores mcpServers config in `mcp-config.json`, not plugin.json

**Location in plan:** Code Design, `MCPConstruct.build_plugin_json`; empirically: `mcp-servers/example-mcp/.claude-plugin/plugin.json`.

**Issue:** The plan writes:
```python
base["mcpServers"] = _load_json(
    self.source_directory / name / "plugin.json"
)["mcpServers"]
```

But the actual example-mcp plugin.json has `"mcpServers": "./mcp-config.json"` — a *path string*, not a dict. The mcpServers content dict is in `mcp-servers/example-mcp/mcp-config.json`. The plan's code would copy the string `"./mcp-config.json"` as the mcpServers value rather than the actual server config. Whether the generated plugin.json should contain the path reference or the inlined dict needs to be decided based on what Claude Code expects.

**Suggested resolution:** Clarify whether MCPConstruct should inline the mcp-config.json content or pass through the path reference as-is. If the Claude Code plugin spec expects a path reference (as the example does), the `build_plugin_json` sketch is wrong and should instead pass through `"./mcp-config.json"` unchanged.

### NICE 4 — Platform `supports` typed as `set[type[Construct]]` but Protocol can't be instantiated — structural typing at class level is invisible to `isinstance`

**Location in plan:** Platform Protocol; DevinPlatform.emit match statement.

**Issue:** `class Construct(Protocol)` — structural subtyping via Protocol means that `SkillConstruct` does NOT explicitly inherit from `Construct`. Python's `isinstance(c, Construct)` will therefore fail at runtime unless `Construct` uses `runtime_checkable`. The `match construct: case SkillConstruct()` patterns in `DevinPlatform.emit` are fine because they match on the concrete class. But the `isinstance(c, construct_cls)` in the Phase 3 loop works because it checks against the *concrete classes* in `supports` — not the Protocol. This is actually correct behavior, but the plan doesn't explain why, leaving the implementing agent confused about why `supports` contains concrete classes not the Protocol.

**Suggested resolution:** Add `@runtime_checkable` to the `Construct` Protocol definition (or add a comment in the plan explaining that `supports` intentionally stores concrete classes, never the Protocol class itself).

### NICE 5 — Plan says "generate_manifest.py --check" for drift detection but current code uses `--check`; plan's new entry point also uses `--check` — verify they're the same flag

**Location in plan:** Success Criteria and Validation Strategy sections; Generator Algorithm entry point.

**Issue:** The existing generator uses `--check` flag (confirmed at line 617 of current `generate_manifest.py`). The plan also uses `--check`. This is consistent. But the current `check_drift` calls `write_all()` internally as its mechanism (regenerating in-place then comparing before/after) — the plan's `_check_drift()` is listed as a function call with no implementation provided.

**Suggested resolution:** Add a skeleton for `_check_drift()` to the plan's Generator Algorithm sketch, even one-line, to confirm the implementing agent understands it should replicate the current snapshot-and-compare logic.
